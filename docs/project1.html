1만 명이 동시에 쿠폰을 요청했을 때, 재고 정합성을 어떻게 지킬 수 있는가. Race Condition을 직접 재현하고, 세 가지 접근으로 해결까지의 사고 과정을 담았습니다.

Architecture

전체 시스템 구조

클라이언트 요청은 API 서버를 거쳐 Redis에서 정합성을 보장하고 DB에 비동기로 저장됩니다. 쿠폰 만료는 Keyspace Notification 이벤트로 처리합니다.

Problem

문제 정의 — 왜 동시성 제어가 필요한가

단건 요청에서는 문제가 드러나지 않습니다. 수백 개의 요청이 밀리초 단위로 동시에 들어올 때만 Race Condition이 발생합니다.

Race Condition

Lost Update 패턴

// 재고 = 1, 두 쓰레드 동시 진입
Thread A: GET remaining → 1
Thread B: GET remaining → 1 ← 같은 값
Thread A: 1 > 0 통과 → decrease()
Thread B: 1 > 0 통과 → decrease()
결과: remaining = -1 (초과 발급)
JMeter 재현 · 재고 100 / 동시 200명

V1 실행 결과

-- 발급 건수
SELECT COUNT(*) FROM coupon_issuance;
→ 200건 (100건 초과 발급)
-- 잔여 재고
SELECT remaining_quantity FROM coupon_event;
→ 80 (0 이어야 정상)
-- remaining=80 의미:
-- 180개 트랜잭션이 서로 덮어쓰기
-- → 차감 결과 유실 (Lost Update)
Thought Process

사고 흐름 — 문제 인식부터 해결까지

각 버전은 이전 버전의 한계를 분석한 결과로 선택했습니다. 더 나은 기술을 찾은 게 아니라, 왜 이 방법이 필요한지를 먼저 이해했습니다.

V1
Step 1 — 문제 재현
동기화 없는 버전 — 일부러 망가뜨리기
"해결책을 바로 적용하기 전에, 문제가 실제로 발생하는지 수치로 확인해야 한다고 생각했습니다. 추측이 아닌 증거가 필요했습니다."

아무 처리 없이 구현 후 JMeter로 동시 200명 요청을 쏘아 Race Condition을 재현했습니다. 이 단계의 목적은 해결이 아니라 문제의 존재를 수치로 증명하는 것이었습니다.

V2
Step 2 — 분산 환경 고려
Redisson 분산 락 — synchronized가 아닌 이유
"처음엔 synchronized 키워드를 쓰면 되지 않을까 생각했습니다. 그런데 서버가 2대 이상이면 JVM이 달라서 의미가 없다는 걸 깨달았습니다. 처음부터 분산 환경을 가정해야 했습니다."
의사결정 근거

synchronized는 단일 JVM 내에서만 동작합니다. Redis 분산 락은 네트워크 너머 모든 서버가 공유하므로 서버 대수와 무관하게 동작합니다. 추가로 락의 생명주기가 트랜잭션을 감싸야 한다는 점 — 커밋 전 락 해제 시 다음 요청이 변경 전 데이터를 읽는 문제 — 을 직접 실험으로 확인했습니다.

Step 3 — 구조적 개선
Lua Script — 락 자체를 없애다
"V2의 2,501ms 응답시간 원인을 분석했습니다. 문제는 '락 설정이 잘못됐다'가 아니라 '락을 기다리는 구조 자체'였습니다. 락 없이도 원자성을 보장할 수 있다면 대기 자체를 없앨 수 있습니다."
의사결정 근거

Redis는 단일 쓰레드로 명령어를 처리합니다. Lua Script는 Redis 내부에서 실행되므로 스크립트 실행 중 다른 명령어가 끼어들 수 없습니다. 락 없이도 원자성이 보장됩니다. 또한 DB 저장을 @Async로 응답 경로에서 분리하여 응답시간을 추가로 단축했습니다.

V4
Step 4 — 트래픽 완충
Sorted Set 대기열 — 급격한 트래픽 흡수
"V3는 빠르지만 극단적 트래픽에서 Redis에도 부하가 집중됩니다. score를 요청 시각으로 쓰면 Sorted Set이 자동으로 선착순을 보장해줍니다."

ZADD로 등록 → 순번 즉시 반환 → 스케줄러가 ZPOPMIN으로 순서대로 꺼내 Lua Script 실행. 트래픽 스파이크를 Redis 레벨에서 흡수합니다.

V5
Step 5 — 라이프사이클 관리
TTL + Keyspace Notification — 만료를 이벤트로
"만료를 스케줄러로 처리하면 매 분마다 전체 DB를 스캔해야 합니다. Redis가 TTL 만료를 직접 알려준다면, 만료된 쿠폰만 정확히 처리할 수 있습니다."

coupon:expire:{issuanceId} 키에 TTL을 설정하면 만료 순간 Redis가 __keyevent@0__:expired 채널에 키 이름을 발행합니다. 리스너는 폴링 없이 이벤트만 수신합니다. value가 아닌 키 이름 자체에 issuanceId를 포함시켜 대상을 식별합니다.

Key Takeaways

핵심 인사이트

01

문제를 먼저 재현한다

해결책보다 먼저, 문제가 실제로 발생하는지 수치로 증명하는 것이 출발점입니다.

02

분산 환경을 전제로 설계한다

단일 서버에서만 동작하는 해결책은 진짜 해결책이 아닙니다. 처음부터 서버 N대를 가정해야 합니다.

03

증상이 아닌 구조를 바꾼다

V2의 느린 응답은 락 설정의 문제가 아니었습니다. 대기 구조 자체를 제거하는 것이 해답이었습니다.
